# 아키텍처 원칙

이 문서는 `nest-pilot-framework`를 기반으로 새로운 백엔드 프로젝트를 시작할 때 따를 기본 설계 원칙을 정의합니다.

## 핵심 방향

이 스타터를 기반으로 만드는 프로젝트는 아래 방향을 기본값으로 삼습니다.

- NestJS 프레임워크 철학
- MSA 지향 서비스 경계
- SOLID 원칙
- TDD 친화적 설계
- Kubernetes 기반 실행 환경 고려
- 실용적인 로컬 개발 경험

## 1. NestJS 철학을 우선한다

- 모듈, provider, DI 구조를 명시적으로 유지합니다.
- Controller는 얇게 유지합니다.
- 비즈니스 로직은 Service 또는 Use Case 성격의 계층에 둡니다.
- 프레임워크가 제공하는 규칙과 흐름을 먼저 따릅니다.
- 불필요한 커스텀 패턴 도입을 최소화합니다.

## 2. MSA 지향 구조를 따른다

- `apps/*` 아래의 각 앱은 독립적으로 이해 가능하고 실행 가능해야 합니다.
- 공통 관심사는 `libs/*`에 둡니다.
- 앱 전용 비즈니스 로직은 해당 앱 내부에 둡니다.
- 공용 라이브러리를 앱 전용 로직 보관소처럼 사용하지 않습니다.
- 서비스 경계를 흐리는 구조를 피합니다.

## 3. SOLID 원칙을 지향한다

### 단일 책임 원칙

각 계층은 역할이 분명해야 합니다.

- controller: 요청/응답, 입력 검증, 서비스 호출
- service/use-case: 비즈니스 규칙과 흐름 제어
- repository: 영속성 처리
- adapter/provider: 외부 시스템 연동

### 개방-폐쇄 원칙

중심 로직을 계속 수정하는 방식보다 확장 가능한 구조를 우선합니다.

### 리스코프 치환 원칙

추상화는 안전하게 대체 가능해야 합니다.

### 인터페이스 분리 원칙

계약은 작고 명확해야 합니다.

### 의존 역전 원칙

인프라 구현체보다 추상화된 경계와 주입 가능한 구조에 의존하도록 설계합니다.

## 4. TDD 친화적인 구조를 지향한다

- 가능하면 전체 앱을 띄우지 않고도 단위 테스트가 가능해야 합니다.
- 비즈니스 규칙은 인프라/전송 계층과 분리합니다.
- 동작이 바뀌면 테스트도 함께 수정하거나 추가합니다.
- 버그 수정 시 가능하면 재현 테스트를 먼저 작성합니다.

## 5. Kubernetes 기반 실행 환경을 고려한다

애플리케이션은 아래 환경에서 실행될 수 있다고 가정합니다.

- 로컬 개발 환경
- 컨테이너 기반 로컬 환경
- Kubernetes 기반 운영 환경

항상 아래를 고려합니다.

- readiness
- liveness
- graceful shutdown
- 시작 시 의존성 준비 상태
- 외부화된 설정
- 가능한 한 stateless 한 서비스 구조

## 6. 로컬 개발 경험도 중요하다

개발자는 아래를 쉽게 할 수 있어야 합니다.

- 특정 앱 하나만 직접 실행
- 필요한 인프라 의존성을 Docker / Docker Compose로 실행
- 설정 파일 위치를 빠르게 이해
- 로컬 머신별 수동 설치를 최소화

## 7. shared library 사용 원칙

`libs/*`는 아래 목적에만 사용합니다.

- 여러 앱이 함께 쓰는 공통 모듈
- 기술적인 공통 관심사
- 안정적으로 재사용 가능한 추상화

앱 전용 도메인 로직을 무분별하게 `libs/*`로 옮기지 않습니다.

## 8. 운영 친화성을 우선한다

아래를 선호합니다.

- 예측 가능한 런타임 동작
- 추적 가능한 로그
- 이해하기 쉬운 설정 구조
- 관측 가능한 실패 방식

아래는 피합니다.

- 숨겨진 마법 같은 동작
- 지나치게 영리한 추상화
- 취약한 부팅 순서 가정
- 내 컴퓨터에서만 되는 방식
